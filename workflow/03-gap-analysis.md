# Step 3 — Gap analysis

Compare the JD keyword list (step 2) against `profile.yaml` experience entries.

## Classify each keyword:

- **Strong match** — directly evidenced in the profile. Specific metric, named tool, documented responsibility. Not inferred — explicitly stated.

- **Partial match** — related experience present but not a direct hit. E.g., JD wants Salesforce, profile has HubSpot. Or JD wants "5+ years in SaaS," profile has 4. These can often be framed honestly without fabrication.

- **Gap** — no meaningful evidence in the profile. Be honest about this; it determines what can and can't be addressed.

## What to do with gaps

Gaps divide into two types:

1. **Addressable gap**: something the user could legitimately claim but hasn't documented in their profile yet. Ask them: "The JD asks for X — do you have experience with that? If so, I'll need a specific example and a number to use it."

2. **Real gap**: something genuinely absent. Don't try to write around it, imply it, or bury it under adjacent claims. Omit it. If the gap is material — if it's a stated must-have and the user doesn't have it — surface it now: "The JD lists X as a requirement and I don't see it in your profile. This is a gap worth knowing about before you apply."

## Internal use

Produce an internal map of strong matches, partial matches, and gaps. You'll use it in:
- Step 4: foregrounding strong-match bullets, de-emphasising low-relevance ones
- Step 5: knowing which keywords to feature in the profile paragraph
- Step 9: flagging open gaps in the output summary

Don't dump the gap analysis on the user as a table unless they ask. Proceed to step 4.
